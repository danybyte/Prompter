# Fix: Retry Button Visible After Successful AI Response

## Problem Analysis

The `showRetry` boolean state (App.tsx:252) controls retry button visibility (App.tsx:1136). It's set to `true` in two places:
- `handleStop` (line 567) — when user manually stops
- Error `catch` block (line 530) — when API call fails

It's set to `false` in two places:
- `handleRefine` start (line 471)
- `handleRetry` start (line 571)

**Root cause:** `showRetry` is never reset on **success**, and it's **global** (not per-session). This causes two bugs:

### Bug A: Stop → Successful response still shows retry
1. User sends message → `showRetry = false`
2. User clicks Stop → `showRetry = true`, but fetch continues in background
3. Fetch succeeds → messages updated, but `showRetry` stays `true`
4. Retry button appears after a **successful** response

### Bug B: Session switch leaks retry state
1. Session A: error occurs → `showRetry = true`
2. User switches to Session B (which has successful responses)
3. `showRetry` is still `true` → retry button shows incorrectly

## Plan

### Fix 1: Reset `showRetry` after successful response
**File:** `src/App.tsx`, inside `handleRefine`'s `try` block (after line 526)

Add `setShowRetry(false)` after `updateSessionWithAIResponse(finalMessages, nextState)`.

### Fix 2: Reset `showRetry` on session switch
**File:** `src/App.tsx`, inside `handleSwitchSession` (line 397-400)

Add `setShowRetry(false)` to clear stale retry state when switching sessions.

### Fix 3: Reset `showRetry` on new session creation
**File:** `src/App.tsx`, inside `handleNewSession` (line 402-422)

Add `setShowRetry(false)` to ensure fresh sessions don't inherit stale retry state.

### Fix 4: Abort fetch on Stop (prevents race condition)
**File:** `src/App.tsx`, refactor `handleRefine` to use `AbortController`

- Create an `AbortController` ref at the component level
- Pass its `signal` to all `fetch` calls in the request chain
- In `handleStop`, call `abortControllerRef.current?.abort()` to actually cancel the in-flight request
- This prevents the race condition where Stop is clicked but the response still updates state

## Files to Modify
- `src/App.tsx` — all fixes are in this single file

## Verification
1. Send a message, wait for AI response → retry button should NOT appear
2. Send a message, click Stop before response → retry button SHOULD appear (correct use case)
3. Send a message, click Stop, but response arrives after stop → retry button should NOT appear (race condition fixed)
4. Trigger error (e.g., invalid API key) → retry button SHOULD appear
5. Click retry → retry button disappears while loading
6. Switch sessions after error in another session → retry button should NOT leak to the other session
